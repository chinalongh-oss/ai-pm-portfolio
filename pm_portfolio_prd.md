# AI 产品经理个人作品集网站 MVP 需求文档

## 1. 网站信息架构 (Information Architecture)
为了在 3 分钟内传递核心价值，网站采用 **“单页长滚动 (Single Page) + 案例弹窗/详情页 (Modal/Detail)”** 的轻量级架构，符合 B 端产品克制、高效的调性。

*   **首页 (Home)**
    *   首屏：个人定位与核心价值主张 (Value Proposition)
    *   能力矩阵：结构化展示核心技能组合
    *   项目大厅：3 个核心商业/AI 项目的入口卡片
    *   关于我：履历摘要与 PDF 简历下载、联系方式
*   **案例详情 (Case Studies) - 独立页或宽体弹窗**
    *   Case 1：AI 工作流与 Agent 架构 (数据质量评估工具) - *展现复杂系统设计与 LLM 落地能力*
    *   Case 2：AIGC 内容智能与提效 (标题/房源生成等) - *展现对生成式 AI 场景的敏锐度与 ROI 把控*
    *   Case 3：商业策略与资源治理 (搜索推荐/定价策略) - *展现深厚的商业底盘与数据驱动能力*

---

## 2. 首页模块设计 (Home Page Modules)
设计风格：B 端 SaaS 官网风格（例如 Stripe, Linear, Vercel 等），深色/浅色冷色调，注重排版与留白，无多余装饰。

1.  **Hero Section (核心定位)**
    *   **Headline**: “懂商业策略与复杂工作流的 AI 产品经理” (或类似表达，强调 Strategy + AI Engineering 的稀缺组合)。
    *   **Sub-headline**: 聚焦于通过 AI 与策略重塑业务流程、提升转化与 ROI。
    *   **CTA (Call to Action)**: [查看核心案例] / [下载完整简历]
2.  **Core Competencies (能力矩阵 - 黄金三角)**
    *   **AI 落地**: 结构化 Agent, LLM 评估, AIGC 场景化。
    *   **策略与商业化**: 搜索推荐, 收益管理 (Yield Management), 定价策略。
    *   **系统架构**: 规则引擎, 审查工作流, 数据质量治理。
3.  **Featured Cases (精选案例卡片)**
    *   按照“业务痛点 -> 核心手段 -> 业务结果”的三段式体现在卡片摘要上。
    *   卡片设计克制，使用抽象的数据图表或系统架构线框图作为配图，摒弃花哨的实物图。

---

## 3. 案例页结构 (Case Study Structure)
面试官/业务负责人看案例的核心是看**“思考逻辑”**和**“解决复杂问题的能力”**，结构需遵循 STAR 法则的变体：

1.  **Summary Header (案例摘要头部)**: 10 秒内了解项目背景与结果。
2.  **The Challenge (业务痛点/挑战)**: 为什么要做这个？当时的业务/数据困境是什么？
3.  **Strategy & AI Solution (策略与 AI 解决方案)**:
    *   为什么选用现在的技术栈（例如为何是 LLM+规则引擎，而不是纯 LLM）？
    *   **核心架构图/流程图**（极其重要，体现 B 端 PM 的抽象能力）。
4.  **Key Implementation (核心落地点/难点攻克)**: 例如如何解决 LLM 的幻觉（通过人工审核流/规则兜底）？如何做数据评估？
5.  **Impact & Metrics (业务收益)**: 具体的 ROI、效率提升百分比、准召率等。
6.  **Learnings (复盘与反思)**: 展现持续迭代的能力与对 AI 边界的客观认知。

---

## 4. 每个案例要展示的字段 (Case Study Fields)
为了保证数据结构的统一，后端/静态数据需包含以下字段：

*   `project_name`: 项目名称（如：基于多模态与 LLM 的房源内容智能生成引擎）
*   `project_role`: 担任角色（如：AI 产品负责人）
*   `context_tags`: 标签组（如：[Agent Workflow], [AIGC], [规则引擎]）
*   `tl_dr`: 核心摘要（一句话总结项目价值）
*   `background`: 业务背景与痛点（控制在 150 字内）
*   `solution_architecture`: 方案架构描述（配合线框图或架构图）
*   `core_features`: 核心功能点（列表形式，如：LLM 评分、模板 SQL 等）
*   `metrics`: 业务指标收益（如：内容生产效率提升 X%，推荐点击率提升 Y%）
*   `artifacts`: 关键产出物截图/示意图（限制 1-2 张，如工作流界面、审计日志面板）

## 5. MVP 范围 (MVP Scope)
**目标：3 天内上线并可用于投递。**

*   **形态**: 响应式（PC/移动端兼容）单页静态网站 (Single Page Application)。
*   **开发技术**: 使用 React/Vue + Tailwind CSS (或直接使用成熟的极简模板修改)，数据全部 Hardcode 在 JSON 文件中，无需后端。
*   **内容就绪**:
    *   首页的文案与 3 个核心项目的 JSON 数据填充。
    *   3 个项目的架构图/流程图（可用 Draw.io 或 Whimsical 画出极简线框图并截图）。
*   **交互**:
    *   平滑滚动 (Smooth scrolling) 到各模块。
    *   点击案例卡片弹出全屏或宽体 Modal 展示详情，避免页面跳转带来的流失感。
*   **部署**: Vercel 或 GitHub Pages，配置自定义域名（可选，但推荐，增加专业感）。

---

## 6. 不做事项 (Out of Scope / Non-Goals)
为了保证 3 天内高标准交付，以下内容坚决不做：

*   **不做花哨的动效**: 拒绝 3D 滚动、复杂的视差滚动 (Parallax)、打字机特效等。保持 B 端产品的严肃性。
*   **不做后台管理系统 (CMS)**: 不需要动态发布能力，更新通过改代码实现。
*   **不夸大 AI 能力**: 描述中避免使用“颠覆”、“AGI”等虚浮词汇，如实描述“规则引擎+LLM”的混合架构，这反而体现了务实的工程化能力。
*   **不做过多的个人生活展示**: 专注职业素养与专业能力，不需要博客文章或日常摄影。

---

## 7. 验收标准 (Acceptance Criteria)
1.  **3 分钟阅读法则**:
    *   **HR 视角**: 在 30 秒内能通过首屏和标签，匹配到 JD 关键词（策略、推荐、AIGC、Agent、数据治理）。
    *   **面试官视角**: 在 2 分钟内阅读完 1 个核心案例，能清晰理解该系统的 Input、Processing、Output 及业务价值。
2.  **视觉克制性**: 全站主色调不超过 3 种（如：黑/白底色，中性灰文字，一种品牌强调色如知乎蓝或深邃紫），页面文字对比度符合 WCAG 标准。
3.  **技术性能**: Google Lighthouse 性能评分 > 90（无大图拖慢加载，首屏秒开）。
4.  **内容自洽**: “商业策略背景”与“AI 落地能力”在案例中自然融合（例如：强调 AIGC 的应用不仅是生成，更是为了提升点击率和 ROI）。
